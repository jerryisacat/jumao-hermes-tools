# 代码文档分层维护规则

类型：`prompt`

## 用途

当一个代码库需要建立、维护或迁移代码结构文档时，用这个 prompt 要求 Agent 按 L0-L4 抽象层级组织文档。它适合避免把架构、设计原则、领域模型、模块细节和运维命令混在同一个扁平文档里。

## 来源与关联

本 playbook 是 [初始化 AGENTS.md 草案](initialize-agents-md.md) 中“代码治理规则 / `CODEGUIDE.md` 规则”的衍生规则。初始化仓库治理文档时，如果需要把 `CODEGUIDE.md` 从“简单代码结构说明”升级为稳定的分层文档体系，可将本 playbook 作为补充 prompt 一并使用。

## 使用方法

把下面的 prompt 粘贴给 Agent，并把 `{project_context}` 替换为目标仓库已有的文档约束、文件路径或你希望补充的项目背景。若目标仓库已经有 `CODEGUIDE.md` 或 `docs/` 文档体系，要求 Agent 先读取现有文档，再按分层规则判断应该新增、迁移或更新哪些文件。

## Prompt

````text
# 代码文档分层维护规则

你维护的代码库采用 L0-L4 分层文档体系。所有代码结构文档必须按抽象层级组织，让读者（人类和 AI）能从鸟瞰逐层下钻，避免把架构、原则、领域模型、文件细节、运维命令混在一起。

{project_context}

## 分层定义

| 层 | 回答的问题 | 内容 | 变更频率 |
|---|---|---|---|
| **L0 系统架构** | "系统长什么样" | 整体架构图、运行时拓扑、仓库布局清单、主干数据流（一张图）、文档优先级表 | 低 |
| **L1 设计原则与边界** | "为什么这样设计" | 分层依赖规则、worker/长任务边界、tenant 权限边界、不可信输入处理链、候选源隔离、反馈影响输出、安全约束 | 低 |
| **L2 领域模型** | "系统处理什么" | 核心实体职责表、状态机与状态转换图、领域术语表 | 中 |
| **L3 模块与调用链** | "具体怎么实现" | 每个包/目录职责、关键文件表、调用链细节、模块级维护规则 | 高 |
| **L4 操作与运维** | "怎么跑起来" | 本地运行命令、验证命令、环境变量全表、部署、测试入口、Docker/本地数据库 | 中 |

## 文件组织规则

L0 + L1 → 合并为一个文件（最稳定，高频阅读）
L2 及之后 → 按模块拆分为独立文件

具体落地（以 TypeScript monorepo 项目为例）：

| 文件 | 层 | 说明 |
|---|---|---|
| `CODEGUIDE.md` | L0 + L1 | 架构图 + 设计原则，单文件。是文档体系入口。 |
| `docs/L2-domain.md` | L2 | 领域模型、状态机、术语表。整体一个领域可单文件；若多领域可拆 `L2-{domain}.md`。 |
| `docs/L3-{module}.md` | L3 | 每个包/app 一个文件：如 `L3-db.md`、`L3-core.md`、`L3-ai.md`、`L3-sources.md`、`L3-web.md`、`L3-worker.md`。 |
| `docs/L4-operations.md` | L4 | 命令、环境变量、部署、测试。可按需拆 `L4-deploy.md`、`L4-testing.md`。 |

`CODEGUIDE.md` 末尾必须包含 L2/L3/L4 的索引段，每项一行摘要 + 链接锚点，让读者能从入口文件下钻到任意模块文件。

## 内容归属规则（改了什么就更新哪层）

| 修改类型 | 更新位置 |
|---|---|
| 新增/删除 package 或 app | L0 仓库布局 + L3 对应模块文件 |
| 新增/修改依赖方向或分层规则 | L1 分层依赖小节 |
| 新增/修改 entity、字段或状态机 | L2 领域模型 |
| 新增/修改关键文件或调用链 | L3 对应模块文件 |
| 新增/修改命令、环境变量、部署配置 | L4 操作运维文件 |
| 安全/边界/不可信输入规则变更 | L1 对应小节 |
| 新增领域概念或术语 | L2 术语表 |

## 稳定性原则

1. **L0/L1 保持稳定。** 不要把文件级细节、具体函数、环境变量值回灌到 L0/L1。它们描述“系统形态”和“为什么”，不描述“怎么实现”。
2. **L3/L4 随实现演进。** 新增文件、改调用链、换命令——直接更新对应 L3/L4 文件。
3. **上层不依赖下层细节。** L0 的架构图不应因为某个文件改名而改动；L1 的原则不应因为某个调用链变化而改动。
4. **下层可引用上层。** L3 模块文件可以引用 L1 的原则编号（如“遵守 L1.4 不可信输入规则”），但 L0/L1 不反向引用 L3 文件名。

## 反模式（禁止）

- ❌ 在 L0 架构图里写具体文件名和函数名
- ❌ 在 L1 设计原则里贴命令或环境变量
- ❌ 把所有层的调用链塞进一张超长 ASCII 图（这是扁平化混层）
- ❌ L2 术语表里写实现细节（如“gravity score 由 `packages/core/src/index.ts` 第 42 行计算”）
- ❌ 修改 L3 文件后不更新 `CODEGUIDE.md` 索引段的摘要
- ❌ 在 L4 里解释架构原理（那是 L0/L1 的事）

## 检查清单（每轮任务结束前）

- [ ] 改了 package/app 结构 → L0 仓库布局 + L3 模块文件已更新
- [ ] 改了依赖方向/边界规则 → L1 已更新
- [ ] 改了 entity/状态机 → L2 已更新
- [ ] 改了文件/调用链 → L3 模块文件已更新
- [ ] 改了命令/环境变量/部署 → L4 已更新
- [ ] L0/L1 未被文件级细节污染
- [ ] `CODEGUIDE.md` 索引段链接指向正确的 L2/L3/L4 文件

## 迁移现有文档时

若现有 `CODEGUIDE.md` 是扁平结构（架构图、数据流、命令、维护规则混在一起），按以下步骤迁移：

1. 提取架构图和拓扑 → L0
2. 提取设计原则、边界规则、安全约束 → L1
3. 提取实体职责表、状态机、术语 → L2
4. 提取每个包/目录的文件表和调用链 → L3（按模块拆文件）
5. 提取命令、环境变量、部署、测试 → L4
6. 在 `CODEGUIDE.md` 重建为 L0+L1 主文件，末尾加 L2/L3/L4 索引段
7. 校验：无信息丢失，只是重新分层归位
````

## 安全 notes

- 使用前先确认目标仓库是否已有文档分层约定；如果已有，优先继承现有命名与入口文件。
- 不要把密钥、生产配置、私有路径或真实部署凭据写入 L4；只记录变量名、占位符和获取方式。
- 迁移现有文档时应先做只读盘点，再提出迁移方案；除非用户明确要求，不要直接大规模改写文档。
- 若目标仓库面向开源发布，L0-L4 文档都应避免包含内部商业策略或非公开部署细节。
